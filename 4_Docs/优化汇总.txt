问题一：重启后 ADB 无法连接  ✅ 已解决
──────────────────────────────────────────────────────────────────────────────

现象:
  板子重启后，adb devices 找不到设备。Windows 设备管理器能看到
  "ADB Interface" (VID_2207 PID_0019) 但状态为 Unknown。

根因:
  Windows 上没有 Rockchip 的 ADB 驱动 (Google USB Driver 未安装)。
  板子重启后，Windows 认出了设备但驱动状态变成 Unknown，ADB 协议栈
  无法通信。

  为什么之前能连: 之前板子没重启过，驱动一直在内存里没掉。一重启
  就暴露了。

  为什么 TCP ADB 能连: 板子起来后 adb 守护进程默认监听 5555 端口，
  即使 WiFi 开了 AP 模式，通过 adb connect <IP>:5555 仍然可以连接。

解决方案:
  方案 A: 安装 Google USB 驱动（一次性根治）
    管理员 PowerShell 运行:
      pnputil /add-driver "C:\Users\wangy\AppData\Local\Temp\google_usb_driver\usb_driver\android_winusb.inf" /install
    安装后重插 USB 线即可。

  方案 B: 启用 TCP ADB 作为备胎
    在 S99hudnavi 启动脚本中增加:
      setprop service.adb.tcp.port 5555
    以后即使 USB 驱动掉了，只要板子网络通就能无线 ADB。

  建议: 两个都做。方案 A 治本，方案 B 兜底。


问题二：重启后 WiFi 未能自动连接（退回了 AP 热点模式）  ✅ 已解决
──────────────────────────────────────────────────────────────────────────────

现象:
  板子重启后 WiFi 地址是 192.168.4.1 (HUD-Setup 热点)，而不是
  10.203.69.x (连接 Xiaomi 14)。手机 App 连不上板子。

根因:
  wifi_setup.py 的 _connect_saved() 失败了，然后直接回退到
  AP 热点模式（192.168.4.1）。失败原因大概率是时序问题——
  S99hudnavi (S99) 启动时 WiFi 内核模块 (S36load_wifi_modules)
  还没完全就绪，wpa_supplicant 关联失败或者 DHCP 拿 IP 超时。

  核心问题: 这段代码在 S99hudnavi 里是一次性的，没有重试:

    _connect_saved()  # wpa_supplicant → udhcpc
      → 失败 → 直接跳到 _start_ap()  # 开了热点就停不下来了
      → main.py 永远启动不了

  当前启动链:
    S99hudnavi start
      → wifi_setup.py (只试一次，失败就开AP永久阻塞)
        → _connect_saved() 失败 → _start_ap() 永远卡住
      → 永远到不了 main.py 启动那一步

解决方案:
  改造 S99hudnavi 启动逻辑:

  1. WiFi 连接和 main.py 启动解耦
     - main.py 不依赖 WiFi 也能跑（GPS/摄像头/HUD 都独立工作）
     - WiFi 连不上只是手机 App 连不上，其他功能正常

  2. WiFi 后台持续重试
     - 如果已有保存的 WiFi 凭据，不要开 AP 热点
     - wpa_supplicant 挂后台持续扫描重连
     - 每 10 秒检测一次，连上就停

  3. 增加 WiFi 模块加载等待
     - S36load_wifi_modules 和 S99hudnavi 之间隔了 60+ 个脚本
     - 但模块实际就绪可能需要额外时间
     - 在 wifi_setup.py 调用前 sleep 3 秒缓冲

  4. 重试次数和超时
     - _connect_saved() 重试 3 次，每次间隔 5 秒
     - 全部失败后再开 AP 热点


问题三：NPU RGA 硬件加速器失效  ✅ 已解决
──────────────────────────────────────────────────────────────────────────────

现象:
  YOLOv5 NPU 推理正常 (55ms/帧)，但检测结果始终为空。
  SDK 自带测试图 out.jpg 之前能检测到公交车+人，现在也检测不到了。
  日志显示:
    RgaBlit(1465) RGA_BLIT fail: Invalid argument
    srect[x,y,w,h] = [0, 0, 0, 0]  ← 图片预处理后的矩形全零

根因:
  RGA (瑞芯微硬件2D加速器) 在图像缩放时失败。
  640x480 → 640x640 letterbox 操作在当前驱动/库版本下不兼容。
  重启后也无法恢复 (非临时性故障，是驱动级bug)。

影响:
  图片无法正确预处理 → NPU 收到垃圾数据 → 检测不到任何物体

解决方案:
  绕过 RGA，改用 CPU 预处理:
    - 640x480 图片 → PIL Image.resize(640,640) → 约 5-10ms
    - 预处理后的 640x640 图片送入 NPU 推理 → 55ms
    - 总耗时约 65ms，帧率从 17fps 降到 15fps
    - 延迟增加可忽略，不产生冲突（CPU预处理和NPU推理是串行的）
    - 修改 ai_detect.py 的 detect_vehicles() 函数

问题三十：PowerShell Set-Content UTF-8 BOM 损坏中文源码 [中危] ✅ 已修复

══════════════════════════════════════════════════════════════
根因分析 (四层问题叠加):
══════════════════════════════════════════════════════════════

[根因A] Mali-G52 GPU page flip 不兼容 Linux 4.19
  - libmali是wayland变体(libmali-bifrost-g52-g2p0-wayland.so)
  - DRM atomic page flip 在4.19内核上对HDMI-A-1返回Invalid argument
  - Weston, SDL2-KMSDRM, EGL均无法通过GPU渲染到屏幕
  - 只有modetest的legacy SETCRTC可以显示(绕过GPU)
  → 解决: 完全绕过GPU, 使用SDL2 dummy软件渲染 + CPU绘制

[根因B] S03weston 开机自启服务抢占DRM
  - /etc/init.d/S03weston 在S99hudnavi之前启动(Debian runlevel顺序)
  - Weston启动后占用DRM framebuffer, 显示"初始化界面"
  - 即使drm_show拿到DRM master, Weston的framebuffer仍可覆盖
  → 解决: mv S03weston → S03weston.disabled + guard.sh启动时killall weston

[根因C] drm_show stderr管道缓冲区泄漏
  - 最初drm_show每帧写"FR\n"到stderr作为帧确认
  - Python端从不读取stderr → 64KB管道缓冲填满 → drm_show阻塞
  → 解决: stderr→subprocess.DEVNULL

[根因D] main.py信号处理导致进程自杀
  - 旧版signal_handler设置_shutdown_flag=True → main.py收到信号就退出
  - drm_show崩溃时pipe写触发SIGPIPE → 杀死main.py → HUD消失
  → 解决: SIGPIPE→SIG_IGN, signal_handler改为pass(guard.sh管理生命周期)

问题三十九：drm_show 连续模式背景闪烁 — 橙蓝交替 [致命] ✅ 已修复
──────────────────────────────────────────────────────────────────────────────
现象: HUD橙色背景与蓝色交替闪烁, modetest测试图案不闪
根因: drm_show连续模式直接在mmap显示缓冲区上读取+RGBA→XR24转换,
      显示控制器在转换过程中扫描到未初始化的buffer(蓝色)或半转换数据
      → 每帧都经历"蓝→橙"的视觉闪烁
诊断: 单帧模式(pause住)完全不闪 → 确认问题在连续循环的buffer操作
解决: 双缓冲架构 — malloc独立后备缓冲区, 读取+转换在后备区完成,
      最后memcpy一次性拷贝到显示framebuffer, 显示控制器永远只看到完整帧
效果: CPU从~90%降至~7%, 零闪烁




问题四十一：S99hudnavi 进程累积永久修复 — 旧guard.sh杀不死 [严重] ✅ 已修复
──────────────────────────────────────────────────────────────────────────────
现象: 重启/调试后 guard.sh + main.py 数量不断累积,
      操作系统killall无法杀死sleep中的shell进程, 导致资源抢占
根因: ① BusyBox killall按进程名匹配(guard.sh), 但进程实际名为/bin/sh, 不匹配
      ② 旧S99hudnavi只读PID文件杀一个进程, 无法清理多次start产生的实例
      ③ 多次调试中反复killall python3+S99hudnavi start造成积压
解决: S99hudnavi重写启动逻辑 —
      ① kill -9 $(cat /tmp/hud.pid) → 杀PID文件记录的进程
      ② ps | grep guard.sh | while read p; do kill -9 $p; done → 扫描所有guard.sh
      ③ killall -9 python3 drm_show weston → 清理子进程
      ④ sleep 2 → 等进程完全退出
      ⑤ start-stop-daemon → 启动唯一新实例
文件: /etc/init.d/S99hudnavi (重写), /opt/hud_navi/guard.sh (简化)
